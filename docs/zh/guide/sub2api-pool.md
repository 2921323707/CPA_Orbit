# Sub2API 订阅号池

CPA Orbit 把“订阅归档、运行网关、Token 使用记录”作为互相关联但权责不同的三层资产。这样既不会让同一份可刷新的 OAuth 凭据长期同时跑在两个网关，也不会用旧归档覆盖 Sub2API 已经刷新的运行状态。

## CPA、Sub2API 与订阅池的关系

| 对象 | 权威方 | 职责 |
|---|---|---|
| 订阅归档 | CPA Orbit 的 `k12/` | 保存原始凭据、入手价、来源和恢复依据 |
| 网关目标 | CPA Orbit 的 `control-plane.db` | 保存 Sub2API/CPA 地址、主备角色、部署默认值和只写密钥 |
| 部署绑定 | CPA Orbit 的 `control-plane.db` | 记录某份归档当前在哪个运行池、由谁管理、期望与实际状态 |
| 运行账号 | Sub2API 或 CPA | 负责调度、Token 刷新、分组、并发和实时额度 |
| 请求明细 | Sub2API | 权威请求日志与计费明细 |
| 用量聚合 | CPA Orbit 的 `control-plane.db` | 保留 15 分钟 Token/成本趋势，最长 90 天 |

推荐关系是：**Sub2API 主池 + CPA 轻量兜底 + Orbit 统一控制面**。主池导入失败时会尝试落到 CPA，但这不是逐请求流量切换。让同一个 refresh token 同时在两边刷新可能互相轮换或失效，因此切换网关必须执行“先撤销源绑定，再部署目标”的显式迁移；目标失败时系统会尝试恢复原绑定。

## 配置 Sub2API

1. 启动 Sub2API，并在其后台生成管理员 API Key。
2. 打开 **号池运维 → 添加 Sub2API**。
3. 填写管理地址和管理密钥。本机部署建议使用 `http://127.0.0.1:8080` 一类回环地址。
4. 按需填写默认分组 ID、账号并发、调度优先级和成本倍率。
5. 设为主网关，保存后点击 **检查连接**。

远程网关必须显式允许并使用 HTTPS。管理密钥是只写字段：只保存在本地，不会通过公开接口回显，也不会进入运维操作日志。

CPA 继续沿用 **设置 → CPA 同步** 中的地址、管理密钥、授权目录和同步开关；控制面里的 CPA 目标只负责启停及主备角色，保持轻量兼容。

## 把 GPT Plus/Codex JSON 转成号池账号

进入 **订阅文件**，保持 **导入后部署到主号池** 开启，选择一份或多份 Codex session JSON：

```text
校验与去重 → 本地归档 → 部署到 Sub2API 主池
                           ↘ 主池失败时尝试 CPA 兜底
```

即使两个运行池都失败，归档也会保留，界面会提示“需要处理运行绑定”。历史归档可在详情抽屉中重新部署。运行在 CPA 兜底池的账号可以一键 **切回主号池**；如果目标部署失败，系统会尝试恢复原绑定。

## 归属与安全删除

- `Orbit 托管`：账号由 Orbit 创建，可以由 Orbit 撤销和删除。
- `外部接管`：账号原本就在外部系统；解绑只清除本地关系，不删除远端账号。
- 删除订阅时先撤销所有活动绑定。若托管的远端账号无法安全删除，本地归档不会被删掉。

## 可视化与 Token 记录

号池运维页展示网关健康、活动绑定、最近部署操作、Sub2API 当前汇总和 7 天 Token 曲线。系统每 5 分钟采集一次，也可以手动刷新；采集失败会把快照标记为陈旧，但继续显示最近一次有效数据。

请求原始明细仍由 Sub2API 保管。Orbit 只保存 15 分钟聚合：请求数、成功/失败数、输入/输出/缓存 Token、平均耗时、首 Token 延迟、标准成本和实际成本。90 天以前的聚合会自动清理。

::: warning 服务条款与封号风险
把订阅账号转换为网关号池可能与上游服务条款冲突，也可能造成封号、服务中断或数据损失。请先确认相关协议与使用授权。Sub2API 官方项目本身也明确提示了同类风险。
:::
